# ضيافتكم — ملفات التسليم (نسخة نهائية v2)

## الملفات
- **diyafatkum_app.html** — النموذج المدموج (عميل + مقدم خدمة + مدير) بشاشة دخول واحدة وبيانات مشتركة. هذا النموذج التجريبي لسه بيانات في الذاكرة (بدون Supabase)، لكن كل حالة وكل انتقال فيه مطابق تمامًا لدوال RPC بقاعدة البيانات — يعني ربطه لاحقًا استبدال، مو إعادة تصميم.
- **schema.sql** — قاعدة بيانات Postgres/Supabase **مُحصّنة (v2)**: كل التعديلات الحساسة (قبول/رفض طلب، دفع، تنفيذ، نزاع، قبول/إيقاف شركة) ما تصير إلا عن طريق دوال RPC مخصصة، مو عن طريق UPDATE مفتوح. مضاف كمان: سجل تتبع تلقائي لكل تغيير حالة (order_status_history)، جدول مدفوعات حقيقي (payments)، جدول تقييمات يحسب متوسط تقييم الشركة تلقائيًا (reviews)، ودعم لأكثر من مستخدم يدير نفس الشركة (company_staff).
- الملفات الثلاثة المنفصلة (customer/provider/admin) صارت غير مطلوبة بعد الدمج، أبقيتها فقط كمرجع.

## ليش هذي النسخة "مُحصّنة" أكثر من الأولى
بالنسخة الأولى، أي مستخدم مسجل دخول (حتى لو مقدم خدمة) كان يقدر تقنيًا يستدعي
`supabase.from('orders').update({status:'paid'})` مباشرة على أي طلب يملك صلاحية قراءته،
لأن صلاحية RLS كانت مفتوحة على مستوى الصف كامل. بهذي النسخة، جدول orders ما فيه صلاحية
UPDATE مباشرة إطلاقًا — لازم تمر من دالة زي `accept_order()` أو `record_payment()`، وكل دالة
تتحقق بنفسها: مين المستخدم؟ يملك هالطلب فعلاً؟ الطلب بالحالة الصحيحة قبل الانتقال؟

## كيف يشتغل الدمج حاليًا (بدون Supabase)
كل الأدوار الثلاثة تشتغل بنفس التبويب (tab) وتتشارك نفس المصفوفات في الذاكرة
(`companies`, `orders`) — يعني إذا بدّلتي الدور من صفحة "حسابي" (بدون تسجيل خروج فعلي)
تقدرين تجربين التدفق الكامل: عميلة ترسل طلب ← تبدلين لمقدم خدمة ← توافقين ← ترجعين للعميلة
وتشوفين النتيجة، كل هذا بذاكرة المتصفح فقط. هذا طبيعي ومتوقع بمرحلة ما قبل Supabase؛
بعد الربط الحقيقي، كل دور بيكون بجلسة/جهاز منفصل والتزامن بيصير عن طريق قاعدة البيانات
+ Supabase Realtime بدل هذا الحل المؤقت.

## خارطة الحقول (Frontend ↔ Database)

| بالنموذج (JS)                          | بالجدول            |
|-----------------------------------------|--------------------|
| `companies[]`                           | `companies`        |
| `company.packages[]`                    | `packages`         |
| `package.tiers[]` (لما `pricingType='tier'`) | `package_tiers` |
| `company.services[]`                    | `services`         |
| `service.tiers[]`                       | `service_tiers`    |
| `orders[]`                              | `orders`           |
| `order.items[]`                         | `order_items`      |
| `currentUser`                           | `profiles` (+ `auth.users`) |

**مهم:** أسعار العناصر بجدول `order_items` تُنسخ (snapshot) وقت إنشاء الطلب، وما ترتبط
حيًا بجدول `packages`/`services`. هذا مقصود — إذا مقدم الخدمة غيّر سعر بكج بعدين، ما يفترض
يتغير سعر طلب قديم خلص وانحسب.

## خطوات الربط الفعلي بـ Supabase (بالترتيب المقترح)

1. **شغّلي `schema.sql`** بمحرر SQL بمشروع Supabase.
2. **فعّلي `pg_cron`** من Database → Extensions إذا ما نجحت `create extension` مباشرة من السكريبت
   (بعض حسابات Supabase تحتاج تفعيلها يدويًا أول مرة).
3. **اربطي Supabase Auth**: بدّلي شاشة "login" الحالية (المبنية بأزرار دخول تجريبي) بنموذج
   `supabase.auth.signInWithPassword()` / `signUp()` حقيقي. جدول `profiles` جاهز يستقبل صف
   تلقائي عند أي تسجيل جديد (عن طريق trigger `on_auth_user_created`)، بس بيصير دوره `customer`
   افتراضيًا — ترقية حساب لـ `provider` أو `admin` لازم تتم يدويًا أو بنموذج طلب انضمام منفصل.
4. **استبدلي المصفوفات المحلية (`companies`, `orders`) باستدعاءات Supabase للقراءة**:
   - `supabase.from('companies').select('*, packages(*, package_tiers(*)), services(*, service_tiers(*))').eq('status','active')`
   - نفس الفكرة لـ orders مع فلترة حسب `customer_id` أو `company_id` حسب الدور.
5. **كل كتابة/تعديل تمر عن طريق RPC بس** (مو `.insert()`/`.update()` مباشرة على الجداول):

   | الإجراء بالواجهة | استدعاء RPC |
   |---|---|
   | عميل يطلب خدمات متفرقة | `supabase.rpc('request_services_order', {p_company_id, p_items, p_notes})` |
   | عميل يطلب بكج | `supabase.rpc('request_package_order', {p_company_id, p_items, p_event_date, p_location, p_guests_expected, p_notes})` |
   | مقدم خدمة يوافق | `supabase.rpc('accept_order', {p_order_id})` |
   | مقدم خدمة يرفض | `supabase.rpc('reject_order', {p_order_id, p_reason})` |
   | تأكيد الدفع | `supabase.rpc('record_payment', {p_order_id, p_method, p_reference})` |
   | مقدم خدمة "تم التنفيذ" | `supabase.rpc('mark_completed', {p_order_id})` |
   | فتح نزاع | `supabase.rpc('raise_dispute', {p_order_id, p_issue})` |
   | المدير يحل نزاع | `supabase.rpc('resolve_dispute', {p_order_id, p_side, p_resolution})` |
   | المدير: قبول/رفض/إيقاف/تفعيل شركة | `supabase.rpc('approve_company' \| 'reject_company' \| 'suspend_company' \| 'reactivate_company', {...})` |

   كل دالة ترجع خطأ واضح بالعربي إذا المستخدم ما يملك صلاحية أو الطلب مو بالحالة الصحيحة —
   لفّي كل استدعاء بـ try/catch وأظهري `error.message` للمستخدم مباشرة.
6. **استبدلي العداد التنازلي والتحديث الحي (`setInterval` بالنموذج) بـ Supabase Realtime**
   subscriptions على جدول `orders`، بدل ما يعتمد كل شيء على تايمر بالمتصفح بس. انتهاء الطلبات
   المعلقة نفسه يصير تلقائي على السيرفر عن طريق `pg_cron` (موجود بالسكيما)، فما يحتاج جافاسكربت
   يعتمد عليه أصلاً.
7. **الصور**: شعارات الشركات حاليًا إيموجي نصي بس (`logo` عمود text). إذا تبين صور حقيقية،
   ضيفي Storage bucket وخزّني الرابط بنفس العمود.
8. **تقييمات العملاء**: بعد إكمال الطلب (`status = 'completed'`)، خلي الواجهة تعرض نموذج تقييم
   يستدعي `supabase.from('reviews').insert(...)` مباشرة (مسموح بس مرة وحدة لكل طلب مكتمل،
   ومحسوب متوسط تقييم الشركة تلقائيًا).

## نقطة يستاهل تنتبهين لها
جدول `orders` بهذي النسخة ما فيه صلاحية UPDATE مباشرة لأي دور — القراءة بس عن طريق RLS،
وكل تغيير حالة يمر إجباريًا عبر دوال RPC (الجدول أعلاه). لو حاولتي `supabase.from('orders').update(...)`
مباشرة من الواجهة، الطلب بيفشل بصمت (٠ صفوف تتأثر) لأنه ما فيه سياسة تسمح بيه أصلاً — وهذا مقصود.

الشي الوحيد المتبقي يستاهل انتباه فعلي: ترقية حساب من `customer` إلى `provider` أو `admin`
لسه يدوية بالكامل (تعديل عمود `role` بجدول profiles مباشرة من لوحة Supabase). إذا تتوقعين
عدد كبير من طلبات الانضمام كمقدمي خدمة، تحتاجين نموذج "التقديم كمقدم خدمة" منفصل يرسل طلب
للمدير بدل ما يكون تعديل يدوي بقاعدة البيانات في كل مرة.
